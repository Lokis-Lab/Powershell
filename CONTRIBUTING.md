# Contributing

Thank you for taking the time to contribute. This document explains how we work together and what we expect from everyone who participates.

## How you can help

- **Report problems** — Open an issue for bugs, regressions, or unclear behavior. Include steps to reproduce, what you expected, and what you observed.
- **Suggest improvements** — Proposals for features or documentation are welcome. A short problem statement and proposed outcome helps reviewers decide quickly.
- **Submit changes** — Pull requests should be focused, described clearly, and kept as small as is reasonable for the change.
- **Improve docs** — Fixes for typos, gaps, and examples are contributions on equal footing with code.

## Before you open a pull request

- **Search existing issues and PRs** to avoid duplicate work.
- **Describe the change** — What problem does this solve? Why this approach?
- **Keep scope tight** — Unrelated cleanups or drive-by refactors make review harder and slow merges.
- **Test your change** — Run the project’s usual checks (tests, linters, builds) when they exist. If the repo has no automation, say how you verified the change in the PR description.
- **Update documentation** when behavior, configuration, or public interfaces change.

## Review and merge

Maintainers review for correctness, maintainability, security, and fit with the project. Feedback is about the work, not the person. You may be asked to adjust implementation details or split large changes. A polite, direct discussion helps everyone move faster.

If a contribution sits without response, a short, friendly bump on the issue or PR is appropriate. Maintainers do their best to respond; silence is rarely intentional.

## Security

Do **not** open public issues for security vulnerabilities. Use the project’s documented security reporting channel if one exists; otherwise contact maintainers privately with enough detail to reproduce and assess impact.
